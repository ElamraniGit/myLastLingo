## 1. Architecture & Project Structure

**Overall:** A genuinely ambitious, well-organized monorepo: FastAPI backend + Next.js 14 (Pages Router) PWA, SQLite persistence, Zustand state, optional local Whisper STT. The layering (`api/ → db/ → services/`, `views/ → hooks/ → lib/`) is sensible and readable.

**Strengths**
- Clear module boundaries; routers split per domain and aggregated in `api/__init__.py`.
- Local-first storage (SQLite) with WAL + sane PRAGMAs.
- Lifespan-based startup, dependency injection of `db_manager` into each router.
- Good prior remediation history (three earlier audit docs in repo) — many earlier bugs already fixed.

**Weaknesses**
- **Global mutable singletons** (`db_manager = None` module-level, `init_api()` injection) make testing and future multi-worker deployment awkward. A FastAPI dependency / app.state pattern would be cleaner.
- **Two `*Config` dataclass definitions** exist for the same concepts (`config/settings.py` *and* duplicated `DictionaryConfig`/`WhisperConfig`/`CacheConfig` inside the service files). Drift risk.
- **Single uvicorn worker assumed everywhere** (in-memory rate limiter, in-memory WS registry, module-level API keys). Fine for the Termux/local target, but it caps scalability and must be documented.
- `run.py` and `backend/main.py` both contain port/host bootstrapping — overlapping responsibilities.

## 2. Bugs, Logic Errors, Edge Cases (found by inspection + execution)

1. **🔴 Fresh-install boot crash** (FIX-CRIT-1) — *fixed & verified*. This alone meant a new user following the README would never get past `./run.py`.
2. **🔴 Data-wipe on update** (FIX-CRIT-2) — *fixed*.
3. **🟠 Whisper fallback fires from inside an `except`** in `transcripts.extract`: when YouTube captions fail, the request returns `processing` and a background task downloads audio. But `_ensure_whisper_dependencies()` raises `HTTPException(503)` only if the deps are missing — good — yet there is **no transcript `status` row**, so the frontend polls `GET /transcripts/{id}` for up to 150s with no way to distinguish "still working" from "failed". (Architectural item A3 from the prior report — still open. See recommendation below.)
4. **🟠 `chat.py` API key is a single shared global** (`_get_groq_key`) stored in `data/.groq_key`. In a multi-user DB, **any user's key is used for every user's chat**, and any user can overwrite it via `/chat/set-key`. Should be per-user (column on `users` or a `user_keys` table).
5. **🟡 `xp.py` streak logic edge case**: streak uses UTC `today`/`yesterday` strings, but `last_active_date` is only updated on `add_xp`. If a user is active across a timezone/midnight boundary (app uses Africa/Casablanca locally but UTC server-side), streaks can break or double-count. Use the user's local date or a stored timezone.
6. **🟡 `_get_recent_conversations` returns messages newest-first** then they're fed back to the model as history in that order in some flows — verify ordering before sending to Groq (history is fetched ASC in `_get_conversation_history`, OK; the `/history` no-id path returns DESC for display, which is fine, but the shape difference is a footgun).
7. **🟡 `videos.delete` deletes `data/downloads/{video_id}.mp4`** but the app never actually downloads MP4s (only metadata + captions). Dead branch / misleading.
8. **🔵 `_estimate_conjugations`** is pure suffix guessing ("go"→"goed"). Acceptable as fallback but will produce wrong forms for irregular verbs; flagged for the education section.

## 3. Performance

**Good**
- `aiosqlite` async I/O (a previously-flagged blocking bug is genuinely fixed).
- Binary-search segment lookup + "fast path" in `useVideoPlayer.getCurrentSegment`.
- TTS disk cache keyed by hash; dictionary results cached in SQLite; SW caches static assets.

**Concerns**
- **🟠 40 ms high-frequency sync interval** in `useVideoPlayer` (`setInterval(…, 40)` = 25×/s) calls `getCurrentTime()` and writes Zustand state continuously while playing. On low-end Android this is real battery/CPU cost. Recommend `requestAnimationFrame` throttled to ~10–15 fps, and only updating the store when the segment/word actually changes.
- **🟠 `get_connection()` opens a brand-new SQLite connection per call** (and sets `PRAGMA journal_mode=WAL` every time). For endpoints that call it 4–8 times (e.g. `/vocabulary/stats`) that's many connection open/close cycles per request. A small connection pool or a single per-request connection passed down would cut this materially.
- **🟡 Minification disabled** (`config.optimization.minimize = false`, `swcMinify:false`) to survive Termux ARM builds. JS payload is larger than necessary (202 kB shared). Reasonable trade-off for the target, but consider building artifacts on a desktop CI and shipping minified bundles.
- **🟡 `framer-motion`** (~40 kB gz) is a heavy dep for a mobile-first PWA where most animations are CSS-friendly.
- **🟡 `/library/sources` merges + sorts both tables in Python then slices** — pagination `total` is computed after loading *all* rows for the user. Fine at small scale, O(n) memory at large.

## 5. Code Quality & Maintainability

- **No automated tests at all** (biggest single quality gap — see §11).
- **Type safety disabled**: `tsconfig` has `strict:false`, `strictNullChecks:false`, and `next.config.js` sets `typescript.ignoreBuildErrors:true` + `eslint.ignoreDuringBuilds:true`. The build is green because checks are *off*, not because the code is type-clean. Re-enable incrementally.
- **`any` everywhere** in `lib/api.ts` (`req<any>`), losing the benefit of the well-defined `types/index.ts`.
- **Duplicated config dataclasses** (noted in §1).
- **Inconsistent datetime handling**: a mix of `CURRENT_TIMESTAMP`, ISO strings, and `YYYY-MM-DD HH:MM:SS`, papered over by a `_normalized_datetime_expr` regex in SQL. Works, but fragile — standardize on UTC ISO-8601 at write time.
- **Large components**: `FlashcardsView.tsx` (548 lines), `LibraryView.tsx` (413), `database.py` (952). Split for readability.
- **Good**: consistent docstrings, clear naming, fix-tags referencing prior bugs.

## 8. Backend / API / Data Model

- **Schema** is reasonable; FKs with cascade; helpful indexes (now correctly ordered).
- **`words` table is global with `UNIQUE(word)`** — a shared dictionary cache across users. Sensible, but means one user's bad/basic entry is served to everyone. Consider separating "canonical dictionary" from "user's saved instance" (already partly done via `saved_words`).
- **Transcripts have no `status`/`error` column** → frontend can't show real progress/failure (A3). Recommend adding `status TEXT` + `error TEXT` and a `GET /transcripts/{id}/status` endpoint.
- **No pagination cap consistency**: some endpoints `le=100`, others `le=200`.
- **`get_video` has no ownership scoping** (any authenticated user can read any video row by id via `/videos/{id}`); only delete checks ownership. Low impact (metadata only) but worth scoping.

## 12. Scalability & Long-term Maintainability

- Designed (correctly) for **single-user-per-device, single-worker**. Multi-tenant/multi-worker would break: in-memory rate limiter, in-memory WS registry, module-global API key, and per-call SQLite connections. If hosting centrally is ever desired: move rate-limit + WS pub/sub to Redis, use a connection pool / Postgres, and per-user secrets.
- SQLite is fine to ~low thousands of words/user; the `words` global table and JSON-in-TEXT columns will eventually want normalization or FTS5 for search.

## 15. Educational / Learning-Science Review

**What's good**
- Real **SM-2-style spaced repetition** with learning steps, lapses, ease factor, and a 4-button (Again/Hard/Good/Easy) Anki-like grading UI. This is the right foundation.
- **Contextual capture**: words are saved with the sentence/source they came from (strong for retention).
- **Multi-modal**: video + synced subtitles + click-to-define + TTS pronunciation + pronunciation trainer + AI chat that knows your weak words. Excellent ingredients.
- **Gamification**: XP, levels, streaks.

**Gaps & evidence-based recommendations**
1. **CEFR accuracy** (now improved): difficulty estimation drives nothing yet — use it to **order new-word introduction** (i+1 comprehensible input) and to flag words above the learner's level.
2. **Active recall variety**: quizzes are mostly recognition (multiple choice). Add **productive recall** (type-the-word, cloze from the original sentence, listening dictation) — production strengthens memory more than recognition.
3. **Interleaving & desirable difficulty**: mix quiz types and old/new words rather than batching; the engine already has the data.
4. **Daily goals & session caps**: `max_reviews_per_day` exists in config but isn't enforced in the UI. Surface a daily goal + "reviews remaining" to leverage streak motivation without burnout.
5. **Leech handling**: words with high `lapses` should trigger reformulation (show a new example/mnemonic), not just reappear — current code surfaces them but doesn't intervene.
6. **Pronunciation feedback loop**: the trainer records audio but (per code) doesn't score against a reference — add phoneme/similarity scoring or at least self-rating that feeds XP.
7. **Listening-first mode**: hide subtitles, then reveal — trains listening (key for comprehension).
8. **Spacing transparency**: show the learner *when* a word is next due and why ("you'll see this in 4 days") — increases trust and metacognition.
9. **Vocabulary breadth analytics**: a CEFR coverage chart ("you know ~60% of A2") is highly motivating and uses data you already have.
